# レッスン（学んだ教訓）

## Emacs Lisp: 文字列とUnicode

- `make-string` はデフォルトでユニバイト文字列を作成する
- Unicode文字（罫線文字 ┌─┐│ 等）を `aset` で書き込むと `Attempt to store non-byte value into unibyte string` エラーになる
- **解決策**: `make-vector` で文字のベクターを使い、`concat` で文字列に変換する
- `(make-string n ?\s t)` は Emacs 28+ で multibyte 文字列を強制するが、Emacs 31 では `Attempt to store non-ASCII char into multibyte string` エラーが発生することがある
- **ベストプラクティス**: グリッド系の処理では文字列ではなくベクターを使う

## 罫線文字の接続方向

- `┌` (corner-tl): 右と下に線が出る
- `┐` (corner-tr): 左と下に線が出る
- `└` (corner-bl): 右と上に線が出る
- `┘` (corner-br): 左と上に線が出る
- 角の文字を決定するとき、周囲のグリッドを読むのではなく、ルーティング方向から直接計算する方が確実

## 描画順序

- 接続線とボックスが重なる場合、描画順序で制御する
- 1. 接続線（線のみ）→ 2. ボックス → 3. 矢印 の順で描画
- 矢印はボックスの辺の上に描画するため、ボックスの後に描く必要がある

## 括弧の不整合

- track-mouse + catch + while のネスト深いコードは括弧の数え間違いが起きやすい
- **ベストプラクティス**: `check-parens` を使って括弧の整合性を確認する
- `batch-byte-compile` のエラーメッセージ `"Invalid read syntax: ")"` は閉じ括弧過多を示す
- `check-parens` の `"Unmatched bracket"` は対応する閉じ括弧が見つからないことを示す

## バッファローカル変数と with-temp-file

- `with-temp-file` は一時バッファで処理を実行するため、元バッファのバッファローカル変数を参照できない
- `ew--boxes` 等の `defvar-local` 変数を `with-temp-file` 内から直接読むとnilになる
- **解決策**: `with-temp-file` の外でデータを読み取ってから、`with-temp-file` 内では書き込みのみ行う
- **ベストプラクティス**: バッファローカル変数を使う場合、一時バッファに入る前にletで値を束縛する

## 全角文字(char-width=2)とグリッド描画

- Unicode罫線文字（┌─┐│└┘▼）は MSゴシック等のCJKフォントで `char-width` = 2（全角幅）
- ASCII罫線（+-|>）は `char-width` = 1（半角幅）
- **問題**: グリッドで1セル=1文字として扱うと、全角文字が画面で2列占めて描画がずれる
- **上辺/中間行の不一致**: 上辺（罫線各2列）と中間行（スペース各1列）で物理幅が異なり矩形にならない
- **解決策**: パディングマーカー方式
  - グリッドを物理列単位で管理
  - `ew--grid-set` で全角文字書き込み時、次のセルにパディングマーカー(`ew--padding`)を配置
  - `ew--render` でパディングをスキップ
  - ボックス描画時、各文字の `char-width` 分ずつ物理列を進める
  - 中間行の内部スペース数 = `width * h_width` （上辺と同じ物理幅に合わせる）
- **マウス座標**: `posn-col-row` は物理列を返すので、物理列グリッドならそのまま使える
- **ラベル**: `string-width` で表示幅を計算、`truncate-string-to-width` でトリム
- **キャンバスサイズ**: 全角罫線時は列数を倍（160列）に拡張する必要がある

## char-width とフォントグリフ幅の不一致

- CJKロケール(ja_JP)ではBox Drawing文字(U+2500-U+257F)の `char-width` が 2 に設定される
- しかしフォントによっては、罫線文字のグリフが実際には半角幅で描画される場合がある
- `char-width` はEmacsの内部カラム計算用であり、フォントの実際のグリフ幅とは独立
- **解決策**: `window-text-pixel-size` でフォントの実ピクセル幅を測定し、半角スペースと比較して判定
- GUIフレームでは `(display-graphic-p)` が t の場合にのみ測定可能
- batchモードでは `char-width` にフォールバック
- **描画関数のステップ幅**: `ew--draw-box` / `ew--draw-h-line` / `ew--draw-connection` では `(char-width ...)` ではなく `(ew--border-h-width)` を使う（実測定値を使用）

## フォントと罫線描画の互換性

- **MSゴシック（等幅）**: 罫線文字を全角（2列幅）で正しく描画。`char-width=2` と一致。パディング方式で正常動作
- **プロポーショナルフォント等**: 罫線文字の幅がフォント依存で異なり、`char-width` と不一致になる場合がある
- **推奨**: MSゴシック等の等幅フォントを使用すること
- **教訓**: 描画崩れの原因がコードのバグかフォント互換性問題かを切り分けるには、まず等幅フォント（MSゴシック）で確認する

## 接続線ルーティングの順方向パターン

- **水平出発→水平到着（right→left等）の順方向**: V→H（2セグメント）を使う
  - 出発辺の外側1セルで即座に垂直に折れ、到着行で水平に進む
  - H→V→H（3セグメント）にするとボックス辺の近くに余分な水平線と角文字が出現してしまう
- **垂直出発→垂直到着（bottom→top等）の順方向**: H→V（2セグメント）を使う
  - 出発辺の外側1セルで即座に水平に折れ、到着列で垂直に進む
  - V→H→V（3セグメント）にするとボックス辺の近くに余分な垂直線と角文字が出現してしまう
- **教訓**: ウェイポイント方式の全面書き換え時に、既存の2セグメント方式を3セグメント方式に変えてしまいデグレを起こした。元の設計意図を理解してから書き換えること
